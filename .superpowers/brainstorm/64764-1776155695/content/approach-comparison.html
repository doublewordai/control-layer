Workloads Feature — Approach Comparison

Two approaches for structuring the Async request view and its relationship to batches

A

Pure Frontend Abstraction

No backend changes. The Async page queries existing batch endpoints, filters for completion_window=1h, then flattens batch results into individual request rows client-side.

Pros

  • Zero backend work — ship faster
  • No new API surface to maintain
  • Batch and Async share the same data layer

Cons

  • Pagination is awkward — batches have N requests each, hard to do flat pagination
  • Filtering/search across requests from different batches is expensive client-side
  • Performance degrades as request count grows
B

Thin Backend Endpoint + Frontend

One new backend endpoint. Add a GET /ai/v1/async/requests endpoint that queries fusillade for individual requests across all 1h batches, with proper server-side pagination, filtering, and search. The Async page calls this directly.

Pros

  • Proper pagination and search from day one
  • Scales to thousands of requests without client-side bottleneck
  • Clean API — frontend just fetches a flat list
  • Can add filtering by model, status, date range easily

Cons

  • Requires a new backend endpoint (moderate effort)
  • Need to query fusillade's request storage directly

💡 Recommendation: Approach B

The flat request list is the core UX of the Async page. Without server-side pagination, it'll break down quickly. The backend work is contained — one endpoint that joins batches → requests with a completion_window filter. This also sets you up for future filtering and search.
